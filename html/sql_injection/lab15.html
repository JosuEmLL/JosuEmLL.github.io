Laboratorio 15

Blind SQLi - tiempo de retardo y extracción

PortSwigger Web Security Academy

Introducción

Este laboratorio combina el concepto ciego anterior. Carece totalmente de retroalimentación visual, no devuelve errores y procesa cualquier cosa igual, por ende, dependemos por completo de retardos de tiempo para extraer la contraseña del administrador.

Desarrollo

En lugar de causar un retraso para confirmar la existencia de la falla, vinculamos el retraso a una comprobación lógica CASE WHEN sobre la contraseña de la base de datos.

Pasos, Pruebas y Payloads

1. Construimos una sentencia de condicional asíncrona dependiente del tamaño y de cada carácter: TrackingId=x'%3BSELECT+CASE+WHEN+(username='administrator'+AND+SUBSTRING(password,1,1)='a')+THEN+pg_sleep(10)+ELSE+pg_sleep(0)+END+FROM+users--.

2. Mandamos la solicitud. Si la respuesta toma 10 segundos, sabemos indudablemente que esa letra que probamos es correcta.

3. Estandarizamos este proceso automatizándolo, asumiendo un riesgo considerable de tiempos de espera largos con scripts o usando macros asíncronas en herramientas como Burp Extender.

Explicación

Usando pg_sleep() condicionalmente logramos forzar a que el hilo ejecutor de la aplicación web retarde su carga única y enteramente cuando la afirmación adyacente es exitosa. Al analizar las métricas de respuesta HTTP extraemos bits de la base de datos.

Reflexiones del Alumno

Este laboratorio me permitió entender la vulnerabilidad de inyección SQL en la cláusula WHERE de una consulta SQL.

Además, aprendí a identificar y explotar vulnerabilidades de inyección SQL mediante la inyección de payloads en parámetros de entrada.

También me ayudó a comprender la importancia de validar y sanitizar entradas de usuario para prevenir ataques de inyección SQL.
